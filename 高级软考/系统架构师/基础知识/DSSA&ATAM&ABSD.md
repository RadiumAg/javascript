## 1. ABSD - 基于架构的软件开发

**ABSD**（Architecture-Based Software Development）是一种架构驱动的软件开发方法，其核心思想是：**软件系统的架构设计不是需求分析完成后才开始的附属活动，而是整个开发过程的起点和核心驱动力**。

### 1.1 核心驱动因素

ABSD 强调由**商业需求**、**质量需求**和**功能需求**三者共同驱动软件架构设计：

| 需求类型 | 描述方式 | 说明 | 示例 |
|---------|---------|------|------|
| **功能需求** (What) | 用例 (Use Case) | 定义系统需要实现的具体业务功能 | 用户可以登录系统 |
| **质量需求** (How Well) | 质量属性场景 | 定义性能、可靠性、安全性等非功能要求 | 10,000 用户同时登录时，响应时间<2 秒 |
| **商业需求** (Why) | 商业目标 | 成本、时间、市场策略、合规性等 | 6 个月内上线，符合 GDPR 规范 |

### 1.2 核心基础

ABSD 方法建立在三个基础之上：

1. **功能分解**：采用模块化思想，将系统功能分解为高内聚、低耦合的构件
2. **架构风格选择**：根据质量需求和商业目标，选择合适的架构风格（分层、微服务、事件驱动等）
3. **软件模板使用**：复用已有的、经过验证的架构模板或软件构件

### 1.3 核心方法论

#### 自顶向下，递归细化
从系统最高层次开始，逐步分解为概念子系统 → 概念构件 → 具体软件构件和类

#### 视角与视图
使用不同视角描述复杂架构，常见视图包括：

- **逻辑视图**：描述系统功能和模块（面向用户和分析师）
- **进程视图**：描述并发和同步行为（面向集成人员）
- **开发视图**：描述代码组织结构（面向开发人员）
- **物理视图**：描述软件部署到硬件的方案（面向运维人员）

#### 并行性
设计活动可与需求分析并行进行，快速建立系统骨架，降低后期返工风险

### 1.4 开发过程（六步流程）

```
1. 架构需求 → 2. 架构设计 → 3. 架构文档化
                                      ↓
6. 架构演化 ← 5. 架构实现 ← 4. 架构复审
```

| 阶段 | 主要活动 | 产出物 |
|-----|---------|--------|
| **1. 架构需求** | 获取商业、质量和功能需求，标识初始构件 | 需求规格说明 |
| **2. 架构设计** | 选择架构风格，映射构件，分析交互 | 架构设计方案 |
| **3. 架构文档化** | 记录设计决策和架构细节 | 《架构规格说明书》《质量设计说明书》 |
| **4. 架构复审** | 邀请外部专家评审架构 | 评审报告 |
| **5. 架构实现** | 编码、构件组装、系统集成 | 可运行系统 |
| **6. 架构演化** | 评估变化影响，更新架构 | 演化后的架构 |

### 1.5 总结

ABSD 是一种以架构为核心、由多维度需求驱动、采用递归细化和视图描述、支持并行开发与持续演进的系统化软件开发方法。

## 2. ATAM - 架构权衡分析法

**ATAM**（Architecture Tradeoff Analysis Method）是由 CMU/SEI 提出的基于场景的、系统化的软件架构评估方法。

### 2.1 核心目标

ATAM 的核心目的**不是判断架构"好"或"坏"**，而是：

- ✅ **识别风险**：找出可能导致质量属性无法满足的架构决策
- ✅ **发现敏感点**：找出对特定质量属性有显著影响的架构决策
- ✅ **明确权衡点**：找出同时影响多个质量属性的决策点
- ✅ **验证业务目标**：确保技术架构能够支撑企业商业目标

### 2.2 关键概念

#### 质量属性效用树（Utility Tree）

ATAM 的起点，将模糊的业务目标转化为具体的、可测试的场景：

```
质量属性 → 细化 → 场景 → (优先级，难度)
例如：性能 → 延迟 → 用户在 2 秒内看到搜索结果 → (高，中)
```

#### 核心术语对比

| 术语 | 定义 | 示例 |
|-----|------|------|
| **场景** (Scenario) | 描述系统在特定条件下运行的故事 | 10,000 用户同时登录 |
| **敏感点** (Sensitivity Point) | 对**某一个**质量属性有显著影响的架构决策 | 加密算法选择影响安全性 |
| **权衡点** (Tradeoff Point) | 同时影响**多个**质量属性的决策点 | 提高加密强度 → 安全性↑ 性能↓ |
| **风险主题** (Risk Theme) | 将相关风险归类以便管理 | 所有性能相关的风险 |

### 2.3 ATAM 评估流程（四阶段）

#### 阶段 1：演示（Presentation）
- ATAM 流程介绍
- 商业动机介绍
- 架构介绍
- 架构风格说明

#### 阶段 2：调查和分析（Investigation & Analysis）
- 生成质量属性效用树
- 分析架构方法
- 识别敏感点和权衡点
- 讨论风险和非风险

#### 阶段 3：测试（Testing）
- 深入分析高风险区域
- 验证假设

#### 阶段 4：报告（Reporting）
- 总结结果
- 展示评估报告

### 2.4 ATAM vs. SAAM

| 特性 | SAAM | ATAM |
|-----|------|------|
| **侧重点** | 主要评估可修改性 | 评估多种质量属性及其权衡 |
| **方法复杂度** | 较简单直接 | 更复杂、更系统 |
| **核心工具** | 场景分析 | 效用树 + 敏感点/权衡点分析 |
| **地位** | ATAM 的前身 | 业界最主流的架构评估方法 |

### 2.5 使用时机

- 🎯 **架构设计初期**：编码开始前验证架构合理性（成本最低）
- 🎯 **架构演进时**：系统重大升级或引入新技术时
- 🎯 **采购评估**：选购第三方系统或框架时

### 2.6 总结

**ATAM = 业务目标 + 质量属性场景 + 架构分析 + 风险/权衡识别**

## 3. DSSA - 特定领域软件架构

**DSSA**（Domain-Specific Software Architecture）是一种面向复用的软件架构方法，核心思想是：**针对特定问题领域，提炼出标准的、可复用的参考架构，支持该领域内新系统的快速、高质量构建**。

### 3.1 核心定义

DSSA 包含三个核心要素：

| 要素 | 说明 | 类比 |
|-----|------|------|
| **领域模型** (Domain Model) | 描述领域内的概念、术语及关系 | 行业知识图谱 |
| **参考需求** (Reference Requirements) | 该领域内所有系统的共性需求 | 标准需求清单 |
| **参考架构** (Reference Architecture) | 标准化的系统结构 | 标准建筑蓝图 |

**目标**：实现"一次设计，多次使用"，降低开发成本，提高质量和一致性

### 3.2 关键特征

- **领域性**：只适用于特定问题域（如"医疗影像系统"）
- **普遍性**：覆盖该领域内绝大多数应用系统的共性需求
- **抽象性**：对具体系统的抽象，限制随意设计发散
- **可复用性**：提供可复用的构件、模式和架构模板

### 3.3 DSSA 分类

| 类型 | 定义 | 特点 | 示例 |
|-----|------|------|------|
| **垂直域** (Vertical Domain) | 针对单一特定系统族 | 深入具体行业，覆盖全套功能 | 航空电子导航系统、银行核心交易系统 |
| **水平域** (Horizontal Domain) | 针对多个领域的共同部分 | 跨行业通用功能，适用面广 | 用户认证系统、日志记录系统、支付网关 |

### 3.4 DSSA 建立过程（五阶段）

```
领域分析 → 领域设计 → 领域实现 → 领域验证 → 领域演化
   ↓          ↓          ↓          ↓          ↓
识别边界   设计架构   开发构件   构建原型   持续更新
```

| 阶段 | 主要活动 |
|-----|---------|
| **1. 领域分析** | 识别领域边界，收集现有系统，分析共性需求和变化点 |
| **2. 领域设计** | 设计参考架构，定义标准组件接口和交互协议 |
| **3. 领域实现** | 开发可复用构件库、框架和工具 |
| **4. 领域验证** | 构建原型系统，验证有效性和覆盖率 |
| **5. 领域演化** | 持续更新和维护 DSSA |

### 3.5 关键角色

- **领域专家**：提供业务知识，界定领域范围
- **领域分析者**：抽取共性需求，建立领域模型
- **领域设计者**：设计参考架构
- **领域实现者**：开发可复用构件

### 3.6 DSSA vs. ABSD

| 特性 | ABSD | DSSA |
|-----|------|------|
| **侧重点** | 单个系统的开发过程 | 一类系统的复用 |
| **驱动方式** | 商业、质量、功能需求驱动 | 领域共性需求驱动 |
| **设计方向** | 自顶向下设计一个系统 | 提取一群系统的共性 |
| **关系** | 具体系统开发方法 | 可被 ABSD 采用的参考架构 |

### 3.7 总结

**DSSA = 领域知识 + 参考架构 + 可复用构件库**

## 4. 4+1 视图模型

**4+1 视图模型**由 Philippe Kruchten 于 1995 年提出，是软件架构领域最经典、应用最广泛的架构描述框架之一。

### 4.1 为什么叫"4+1"？

模型包含 5 个视图，其中**4 个是主要的结构视图**，**1 个是贯穿始终的场景视图**：

```
┌─────────────────────────────────────────────────┐
│              场景视图 (+1)                       │
│           (驱动和验证其他四个视图)                │
└─────────────────────────────────────────────────┘
           ↓         ↓         ↓         ↓
    ┌──────────┬──────────┬──────────┬──────────┐
    │ 逻辑视图  │ 开发视图  │ 进程视图  │ 物理视图  │
    │          │          │          │          │
    │ 功能需求  │ 代码组织  │ 并发行为  │ 部署方案  │
    │ 用户/分析师│ 程序员   │ 集成人员  │ 运维人员  │
    └──────────┴──────────┴──────────┴──────────┘
```

### 4.2 五个视图详解

| 视图 | 关注点 | 主要涉众 | 描述内容 | 核心问题 |
|-----|--------|---------|---------|---------|
| **逻辑视图** | 系统功能需求 | 最终用户、功能分析师 | 类图、对象图、状态图、交互图 | 系统"做什么"？ |
| **开发视图** | 软件模块组织 | 程序员、项目经理 | 组件图、包图、目录结构、分层架构 | 系统如何"构建"？ |
| **进程视图** | 运行时行为 | 系统集成人员、性能工程师 | 进程/线程图、通信机制、死锁检测 | 系统如何"运行"？ |
| **物理视图** | 软件到硬件映射 | 系统运维人员、部署工程师 | 部署图、网络架构图、节点配置 | 系统"部署"在哪里？ |
| **场景视图** (+1) | 用例和场景 | 所有涉众 | 用例描述、用户交互流程 | 如何验证一致性？ |

### 4.3 场景视图（+1）的作用

- 🎯 **驱动设计**：其他四个视图的设计必须支持场景实现
- 🎯 **验证一致性**：检查四个视图之间是否存在矛盾
- 🎯 **沟通桥梁**：让非技术人员也能理解架构

### 4.4 核心价值

1. **分离关注点**：不同角色只看自己关心的视图，降低认知负荷
2. **全面性**：覆盖从功能、代码、运行到部署的全生命周期
3. **一致性验证**：通过场景视图确保四个静态视图协调一致

### 4.5 总结

4+1 视图模型是软件架构师的"瑞士军刀"，告诉我们：描述架构不能只画一张图，而要根据受众不同，提供四张蓝图，并用场景将它们串联起来验证。

## 5. Web 服务协议对比

| 协议/标准 | 全称 | 作用 | 是否负责通信 |
|----------|------|------|------------|
| **UDDI** | Universal Description, Discovery and Integration | 基于 XML 的注册和发现机制，用于发布和查找 Web 服务 | ❌ 否 |
| **WSDL** | Web Services Description Language | 描述 Web 服务接口和操作的 XML 语言，定义输入、输出和调用方式 | ❌ 否 |
| **SOAP** | Simple Object Access Protocol | 基于 XML 的通信协议，用于在网络上传递 Web 服务请求和响应 | ✅ **是** |
| **BPEL** | Business Process Execution Language | 业务流程执行语言，用于组合多个 Web 服务 | ❌ 否 |

### 典型考题

**题目**：以下哪个协议用于实现 Web 服务提供者与消费者之间的远程调用？

- A. UDDI - 注册和发现机制，不负责远程调用
- B. WSDL - 描述服务接口，不负责实际通信
- **C. SOAP - 通信协议，实现远程调用 ✅**
- D. BPEL - 业务流程组合，不用于远程调用

**答案：C**
